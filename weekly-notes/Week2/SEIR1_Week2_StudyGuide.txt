SECTION 1: THE TWO PLANES
--------------------------

Core Statement (Memorize This)

    Microsoft Entra ID is the trust plane — identity authority lives here.
    These are two separate systems connected by deliberate configuration.

SECTION 2: THE TRUST FLOW
--------------------------

The Flow (Memorize This)
    User → Identity Provider → Cloud Resource

What happens at each step:
    User: Tries to access a GCP resource.
    Identity Provider (Entra ID): Authenticates the user. Issues a token.
    Cloud Resource (GCP): Receives the token. Checks authorization. Grants or denies access.

Key Point
    GCP never authenticates the user directly.
    It delegates authentication to Entra ID.
    GCP only handles authorization.

    Where does authentication happen in this flow?
    Where does authorization happen?
    What does GCP receive from Entra ID?
    What happens if the identity provider is unavailable?

SECTION 3: WHAT IS A TRUST BOUNDARY
--------------------------------------

Definition
    The line between what is inside your trusted system and what is outside.
    Anything crossing the boundary must present verified identity.

What Crosses the Trust Boundary
    A token — a signed document issued by Entra ID that says:
    "This is who the user is and what claims they carry."
    GCP accepts the token because it trusts Entra ID to issue them correctly.

What Happens When the Boundary Breaks
    If the trust configuration between Entra ID and GCP breaks:
    Tokens are no longer accepted.
    Users cannot access GCP resources.
    The infrastructure is still running. Access is gone.

Check Yourself
    What is a trust boundary in your own words?
    What crosses the trust boundary between Entra ID and GCP?
    What breaks when the trust boundary is misconfigured?

SECTION 4: FEDERATION
----------------------

What It Is
    The mechanism that extends trust from one system to another.
    Entra ID federates into GCP so Microsoft-authenticated users
    can access GCP resources without a separate GCP account.

Why It Matters
    Without federation, every cloud needs its own user accounts.
    With federation, one identity (in Entra ID) works across all federated systems.

Why It Is Fragile
    Federation depends on a configured trust relationship.
    If a certificate expires, the configuration drifts, or a policy changes —
    the federation breaks and everyone who depends on it loses access simultaneously.

Check Yourself
    What problem does federation solve?
    What makes federation fragile?
    If federation breaks at 3am, who loses access?

SECTION 5: CLOUDS ARE NOT JUST COMPUTE
----------------------------------------

The Wrong Mental Model
    "The cloud is just someone else's computer."
    This model treats the cloud as infrastructure only.
    It ignores the trust plane entirely.

The Right Mental Model
    The cloud is a trust boundary.
    Every resource sits behind an identity checkpoint.
    Access is never assumed. It is always verified.
    The identity plane controls the resource plane.

Why This Matters
    Engineers who treat cloud as just compute
    make identity decisions carelessly.
    They create service accounts with too much access.
    They skip federation configuration.
    They ignore blast radius.
    Those engineers cause the most expensive incidents.

Check Yourself
    Why is "the cloud is just someone else's computer" a dangerous mental model?
    What does it mean that identity controls the resource plane?
