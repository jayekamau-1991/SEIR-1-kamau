## 3. Lab Overview

Lab 2 reproduces the Lab 1 VM deployment using Terraform instead of manual click-ops in the GCP Console. This is the real transition from manual infrastructure to infrastructure as code — reproducible, version-controlled, and reviewable.

The same SEIR-I Ops Panel from Lab 1 is deployed via a Terraform startup script. The gate script validates the same endpoints proving the infrastructure is identical regardless of how it was deployed.

> "Click-ops gets you started. Terraform gets you hired."

## 6. Steps to Complete the Lab

### Step 1 — Set Up Service Account Credentials
```bash
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/your/service-account-key.json"
```

### Step 2 — Initialize Terraform
```bash
cd weekb/terraform
terraform init
```

Expected output:
```
Terraform has been successfully initialized!
```

### Step 3 — Validate the Configuration
```bash
terraform validate
```

Expected output:
```
Success! The configuration is valid.
```

### Step 4 — Plan the Deployment
```bash
terraform plan -var="project_id=YOUR-PROJECT-ID" -out tfplan 2>&1 | tee plan.txt
```

Review the plan — it should show 2 resources to create:
- `google_compute_firewall` — allows port 80
- `google_compute_instance` — the VM running the ops panel

### Step 5 — Apply the Deployment
```bash
terraform apply tfplan 2>&1 | tee apply.txt
```

### Step 6 — Get the VM URL
```bash
terraform output vm_url
terraform output vm_external_ip
```

### Step 7 — Verify the Deployment

**Browser proof:**
Open `http://YOUR-EXTERNAL-IP/` in your browser.

**Machine proof:**
```bash
curl -s http://YOUR-EXTERNAL-IP/healthz
```

**Engineer proof:**
```bash
curl -s http://YOUR-EXTERNAL-IP/metadata | jq .
```

### Step 8 — Run the Gate Script
```bash
cd weekb/python
chmod +x gate_lab2_http.sh
VM_IP=YOUR-EXTERNAL-IP ./gate_lab2_http.sh
```

Gate script source:
https://github.com/BalericaAI/SEIR-1/blob/main/weekly_lessons/weekb/python/gate_lab2_http.sh

### Step 9 — Collect Artifacts
- `gate_result.json` — generated by gate script
- `badge.txt` — generated by gate script
- Screenshot of homepage in browser
- `plan.txt` — saved from Step 4
- `apply.txt` — saved from Step 5

## 8. Teardown

### Stop the VM (preserve data)
1. GCP Console → Compute Engine → VM Instances
2. Select `chewbacca-node-lab2`
3. Click **Stop**

### Destroy with Terraform (full teardown)
```bash
cd weekb/terraform
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/your/service-account-key.json"
terraform destroy -var="project_id=YOUR-PROJECT-ID"
```

Type `yes` to confirm. This removes:
- The VM instance
- The firewall rule
- All associated resources

> **Note:** Always destroy or stop your VM after completing the lab to avoid unexpected charges.

## 9. Lessons Learned

### What is relatable to the user/customer?
Terraform makes infrastructure reproducible. The same VM that took 10+ manual clicks in Lab 1 is now deployed with a single command. This is how real engineering teams manage cloud infrastructure at scale — version controlled, reviewable, and consistent across environments.

### Struggles encountered
- **Dollar signs in startup scripts** — Terraform interprets `${}` as variable interpolation inside heredoc strings. All bash variables inside the startup script required escaping from `${}` to `$${}`. This caused `terraform init` to fail until the fix was applied.
- **Stale plan error** — After the VM state changed, the saved `tfplan` file became stale. A fresh `terraform plan` was required before applying again.
- **Ephemeral IP changed** — The VM's external IP changed after a stop/start cycle. Always retrieve the current IP using `terraform output vm_external_ip` or from the GCP Console before running the gate script.
- **Missing credentials** — Terraform requires GCP credentials set via `GOOGLE_APPLICATION_CREDENTIALS` environment variable when not using `gcloud auth application-default login`.

### How did you save money after teardown?
- Stopped the VM immediately after collecting all artifacts
- Verified no NAT Gateways or Load Balancers were left running
- Monitored GCP Billing → Budgets & Alerts
- Used `terraform destroy` to cleanly remove all resources when fully done

## 11. Troubleshooting

### terraform init fails — configuration errors
```bash
# Check for dollar sign escaping issues in startup script
# Change ${ to $${ inside Terraform heredoc strings
terraform validate
```

### Saved plan is stale
```bash
# Re-run plan before applying
terraform plan -var="project_id=YOUR-PROJECT-ID" -out tfplan
terraform apply tfplan
```

### No credentials found
```bash
# Set credentials via environment variable
export GOOGLE_APPLICATION_CREDENTIALS="/path/to/key.json"

# Or authenticate via gcloud
gcloud auth application-default login
```

### External IP is empty after apply
```bash
# Get current IP from GCP Console or via gcloud
gcloud compute instances describe chewbacca-node-lab2 \
  --zone=us-central1-a \
  --format='get(networkInterfaces[0].accessConfigs[0].natIP)'
```

### Gate script permission denied
```bash
chmod +x gate_lab2_http.sh
```

### Homepage not loading after VM starts
```bash
# SSH into VM and check nginx
systemctl status nginx --no-pager
sudo systemctl restart nginx
```
